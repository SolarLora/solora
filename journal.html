problems encountered + solutions
✕
Deduplication dropping all heartbeats after first packet
Sequence-number approach used buf[3] as the sequence ID. In HeartbeatPacket buf[3] is the role field — always 0, 1, or 2. Every heartbeat from the same device had the same sequence ID so all were dropped as duplicates after the first.
// Bug — role field used as sequence number uint8_t seq = (len > 3) ? buf[3] : 0; if (alreadySeen(orig, seq, type)) return;
Solution: rewrote deduplication as a time-based 25 second window per origin+type. No sequence number needed — if the same device type of packet arrives within 25s it's a duplicate. After 25s it's a new packet and gets relayed.
// Fix — time window, no sequence number if (now - seen[i].seenAt < 25000UL) return true; // drop duplicate
root cause found resolved mesh
✕
Dual coordinator on boot — both devices promoting simultaneously
lastCoordHeard initialized to 0. On boot millis() - lastCoordHeard was a huge number immediately triggering the failover check and promoting both devices to coordinator before they had a chance to hear each other.
Solution: initialize lastCoordHeard = millis() in setup(). Devices now listen for the full 10 second role determination window before any failover logic runs.
boot timing bug resolved mesh
✕
HeartbeatPacket struct size changed — broke mesh communication
Removed the localHour field from HeartbeatPacket when time sync was stripped out. This changed the struct size from 25 to 24 bytes. The len >= sizeof(HeartbeatPacket) check on receivers failed silently — all heartbeats dropped.
Solution: replaced localHour with a reserved uint8_t field that keeps the struct size stable. Added comment warning not to remove it.
uint8_t reserved; // keeps size stable — do not remove
struct alignment bug resolved firmware

what was learned
L
Packed structs are fragile — any size change breaks the mesh
When every device on the mesh uses sizeof() to validate incoming packets, a one-byte change anywhere in a struct silently breaks all communication between old and new firmware. Lesson: never remove fields from packed structs — mark unused fields as reserved instead. Always flash all devices with the same firmware version.
L
Time-based deduplication is more robust than sequence numbers for LoRa
Sequence numbers only work reliably when packet fields are stable and well-defined. LoRa mesh packets share a common 3-byte header but the 4th byte means something different in each packet type. A time window deduplicator works purely on origin ID and packet type — no field misinterpretation possible.
L
WiFi mode off saves more power than any other single change
WiFi radio in idle/scanning state draws ~80-100mA even when not connected or transmitting. Calling WiFi.mode(WIFI_OFF) after role assignment on workers was the single biggest power saving — more than CPU frequency reduction and LoRa power combined. Critical for solar operation.
decisions made
D
MAX_HOPS = 3 as the permanent setting
Evaluated tradeoffs between range and collision risk. At 3 hops: ~45 mile range, low collision probability, stays well within LoRa 1% duty cycle limits. At 6 hops: ~90 miles but exponentially increasing duplicate cascade risk. 3 hops is the right balance for a consumer product.
D
Time sync via LoRa removed — simplifies firmware significantly
NTP time sync via LoRa added complexity, a new packet type, timezone handling, and a dependency on the coordinator having internet. Removing it entirely and replacing with a simple 2-minute screen timeout achieved the same power saving goal without any of the complexity.
★
Product named Solora — Solar + LoRa
Evaluated: SolarBit (taken — BitAxe project), ROAM (taken — Solana/Samsung WiFi network), LoraCoin (dead token, 2 holders), MESHSAT, GRIDLESS, DRIFT. Solora won: instantly understood, globally pronounceable, sounds like hardware not a coin, available on .network TLD which perfectly describes the product.
D
Each device mines with owner's own Bitcoin address — never ours
Rejected Option B (all devices point to our address, we distribute). Creates money transmission liability, securities law exposure, and dependency on us forever. Correct architecture: each customer enters their own BTC address at setup. Mining pool pays them directly. We never touch the money at any point.

reflection
What went well?
The deduplication rewrite was the key unlock — once that was fixed the mesh became reliable and everything else built on top of it cleanly. The decision to remove time sync entirely rather than patch it was the right call. Simpler is more reliable. The brand and website coming together in a single session gave the project a real identity it lacked before.
What was the hardest problem?
The struct size bug was the most subtle. The firmware compiled without errors, uploaded fine, and appeared to run correctly — but the mesh was silently dropping all heartbeats. Finding it required going back through all the changes that had been made and counting bytes manually. Key lesson: when mesh communication breaks after a firmware change, check struct sizes first.
What would you do differently?
Should have locked the packet struct sizes earlier with a static assert. In C++ you can add compile-time verification: static_assert(sizeof(HeartbeatPacket) == 25, "HeartbeatPacket size changed"). This would have caught the byte size change immediately at compile time rather than at runtime debugging.
Biggest insight of the day?
The product philosophy — "the device works for its owner, not for its maker" — emerged organically from a series of individual technical and legal decisions. It wasn't planned. By the end of the day every decision made independently pointed back to the same principle. That consistency is what makes the product coherent.
